# 文書番号採番仕様

## フォーマット

```text
{文書種別コード}{部署コード}-{YYMM}{連番}
```

| フィールド     | 例     | 説明                                                                                                          |
| -------------- | ------ | ------------------------------------------------------------------------------------------------------------- |
| 文書種別コード | `内`   | 文書種別のコード（`document_kinds.code`）。最大 2 文字                                                        |
| 部署コード     | `設計` | 業務種別の担当部署コード（`departments.code`）。最大 10 文字                                                  |
| `-`            | `-`    | 固定区切り文字                                                                                                |
| YYMM           | `2603` | 西暦年下 2 桁 + 月 2 桁（登録年月）                                                                           |
| 連番           | `001`  | 同一 `{文書種別コード}{部署コード}{YYMM}` 内の通し番号。文書種別ごとに桁数が異なる（2 または 3 桁、ゼロ埋め） |

```text
内設計-2603001   # 社内、設計部署、2026年3月、1件目（3桁連番）
内設計-2603002   # 社内、設計部署、2026年3月、2件目
議保全-260301    # 議事録、保全部署、2026年3月、1件目（2桁連番）
```

---

## 採番手順

文書登録リクエストを受けた際、以下の順序で採番する。

1. リクエストから `project_id` と `doc_kind_id` を取得する
2. `projects` テーブルから `discipline_id` を取得する
3. `disciplines` テーブルから `department_id` を取得する
4. `departments` テーブルから `code`（部署コード）を取得する
5. リクエストから `doc_kind_id` を取得し、`document_kinds` テーブルから `code`, `seq_digits`（連番桁数）を取得する
6. 登録日時から `YYMM` を生成する（サーバー時刻の UTC → JST 変換後の年月）
7. `documents` テーブルで `doc_number LIKE '{文書種別コード}{部署コード}-{YYMM}%'` となる最大連番を取得する
8. 連番をインクリメントし、`seq_digits` 桁にゼロ埋めして文書番号を生成する
9. 生成した文書番号を `doc_number` に、部署コードを `frozen_dept_code` に、文書種別を `doc_kind_id` にセットして保存する

### 採番のアトミック性

同一の `{文書種別コード}{部署コード}{YYMM}` に対して同時採番が発生した場合に重複しないよう、採番処理全体をトランザクション内で実行し、`documents` テーブルに対して `SELECT ... FOR UPDATE` または `SERIALIZABLE` 分離レベルを使用する。

`documents.doc_number` の `UNIQUE` 制約を最終防衛線とし、競合で `409 CONFLICT` が発生した場合は同一トランザクション境界で最大 3 回まで再試行する。

---

## 凍結ルール

採番後の `doc_number` および `frozen_dept_code` は**変更不可**とする。

- `PUT /api/v1/documents/:id` では `doc_number`, `frozen_dept_code` を更新対象から除外する
- アプリケーション層でこれらのフィールドを無視し、DB に反映しない

---

## 部署再編時の挙動

### 部署廃止・統合

- 廃止部署に紐づく業務種別の `department_id` を新しい担当部署に更新する
- 廃止部署のコード（`departments.code`）は**再利用しない**
- 過去に採番済みの文書番号は変更しない（`frozen_dept_code` が凍結済みのため）
- 廃止部署の有効期間を `effective_to = 廃止日` に更新する

### 採番への影響

| タイミング           | 挙動                                               |
| -------------------- | -------------------------------------------------- |
| 廃止前に登録した文書 | 旧部署コードが `frozen_dept_code` に残る。変更なし |
| 廃止後に登録した文書 | 新担当部署のコードで採番される                     |

### 連番の継続性

部署コードが変わった場合（例: `設計` → `開発`）、連番は新しい部署コードで 1 から始まる。
旧コードの連番と混在することはない。

---

## 業務種別コードの管理

`disciplines.code` は業務種別マスタに登録する際に決定する。
一度登録したコードは変更しない。

現在想定している業務種別コード例:

| コード | 業務種別名 | 担当部署例 |
| ------ | ---------- | ---------- |
| `CAD`  | CAD 設計   | `設計`     |
| `FEM`  | FEM 解析   | `解析`     |
| `MECH` | 機械補修   | `保全`     |
| `ELEC` | 電気工事   | `電気設備` |

業務種別は「どの業務プロセスか」という分類であり、担当部署と紐づく。

---

## 部署コードの管理

`departments.code` は部署設立時に決定する。廃止後も再利用しない。

現在想定している部署階層:

```text
役員
├─ 技術
     ├─ 設計
     ├─ 開発
     └─ 解析
├─ 運営
     ├─ 保全
     └─ 電気設備
└─ 総務
```

| コード | 部署名   | 親部署コード |
| ------ | -------- | ------------ |
| `役員` | 役員     | -            |
| `総務` | 総務     | `役員`       |
| `技術` | 技術     | `役員`       |
| `設計` | 設計     | `技術`       |
| `開発` | 開発     | `技術`       |
| `解析` | 解析     | `技術`       |
| `運営` | 運営     | `役員`       |
| `保全` | 保全     | `運営`       |
| `電設` | 電気設備 | `運営`       |

文書番号に現れる部署コードは、業務種別に担当部署として紐づいた末端部署のコードである。親部署（役員・技術・運営）は直接業務種別を持たない限り文書番号には現れない。

---

## 文書種別の管理

文書種別は「文書の性質・用途」による分類であり、業務種別とは独立した概念。
`document_kinds.code` は登録時に決定し、変更しない。

| コード | 文書種別名 | 連番桁数 |
| ------ | ---------- | -------- |
| `契`   | 契約       | 3        |
| `提`   | 提出       | 3        |
| `内`   | 社内       | 3        |
| `入`   | 入手       | 3        |
| `品`   | 品質       | 3        |
| `議`   | 議事録     | 2        |
| `稟`   | りん議     | 3        |
| `教`   | 教育記録   | 3        |
