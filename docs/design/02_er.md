# ER 図

Mermaid による Entity-Relationship 図。実装の直接の根拠とする。

## ER 図

```mermaid
erDiagram
    departments {
        uuid id PK
        varchar code UK "採番用コード（不変）"
        varchar name
        uuid parent_id FK
        date effective_from
        date effective_to "NULL = 現在有効"
        uuid merged_into_id FK
        timestamptz created_at
        timestamptz updated_at
    }

    employees {
        uuid id PK
        varchar name
        varchar employee_code UK "社員番号"
        varchar ad_account UK "ADアカウント"
        varchar role "admin|project_manager|general|viewer"
        boolean is_active
        timestamptz created_at
        timestamptz updated_at
    }

    employee_departments {
        uuid id PK
        uuid employee_id FK
        uuid department_id FK
        boolean is_primary "主務フラグ。同一有効期間内で1件のみtrue"
        date effective_from
        date effective_to "NULL = 現在所属"
        timestamptz created_at
    }

    disciplines {
        uuid id PK
        varchar code UK "業務種別コード（採番用）"
        varchar name
        uuid department_id FK "担当部署"
        timestamptz created_at
        timestamptz updated_at
    }

    projects {
        uuid id PK
        varchar name
        varchar status "planning|active|completed|cancelled"
        date start_date
        date end_date
        varchar wbs_code UK "SAP WBSコード（参照用）"
        uuid discipline_id FK
        uuid manager_id FK
        timestamptz created_at
        timestamptz updated_at
    }

    documents {
        uuid id PK
        varchar doc_number UK "採番済み文書番号（凍結）"
        int revision "文書改版番号（1始まり）"
        varchar title
        varchar file_path
        uuid author_id FK
        uuid doc_kind_id FK
        varchar frozen_dept_code "採番時の部署コード（凍結）"
        varchar confidentiality "public|internal|restricted|confidential"
        varchar status "draft|under_review|approved|rejected"
        uuid project_id FK "NOT NULL"
        timestamptz created_at
        timestamptz updated_at
    }

    tags {
        uuid id PK
        varchar name UK
    }

    document_tags {
        uuid document_id FK
        uuid tag_id FK
    }

    approval_steps {
        uuid id PK
        uuid document_id FK
        int route_revision "承認ルート版（1, 2, 3...）"
        int document_revision "ルート設定時の文書改版番号（凍結）"
        int step_order "承認順序（1, 2, 3...）"
        uuid approver_id FK
        varchar status "pending|approved|rejected"
        timestamptz approved_at
        text comment
        timestamptz created_at
    }

    document_kinds {
        uuid id PK
        varchar code UK "文書種別コード（不変）"
        varchar name
        int seq_digits "採番連番桁数"
        timestamptz created_at
        timestamptz updated_at
    }

    document_registers {
        uuid id PK
        varchar register_code UK "系列コード（例: 契設計）"
        uuid doc_kind_id FK
        uuid department_id FK
        varchar file_server_root "ファイルサーバルートパス"
        varchar new_doc_sub_path "新規登録時サブパステンプレート"
        varchar doc_number_pattern "既存文書スキャン用正規表現"
        timestamptz created_at
        timestamptz updated_at
    }

    path_scan_issues {
        uuid id PK
        uuid document_id FK "NULL = 未マッチ"
        varchar found_path
        varchar issue_kind "no_match|multiple_match"
        timestamptz resolved_at "NULL = 未解決"
        timestamptz created_at
    }

    departments ||--o{ departments : "parent_id（階層）"
    departments ||--o{ employee_departments : ""
    employees ||--o{ employee_departments : ""
    departments ||--o{ disciplines : "担当部署"
    disciplines ||--o{ projects : ""
    employees ||--o| projects : "manager_id"
    projects ||--|{ documents : ""
    employees ||--o{ documents : "author_id"
    document_kinds ||--o{ documents : "doc_kind_id"
    documents ||--o{ document_tags : ""
    tags ||--o{ document_tags : ""
    documents ||--o{ approval_steps : ""
    employees ||--o{ approval_steps : "approver_id"
    document_kinds ||--o{ document_registers : ""
    departments ||--o{ document_registers : ""
    documents ||--o{ path_scan_issues : ""
```

## 主要な設計上の判断

### 部署の有効期間モデル

`is_active` フラグではなく `effective_from / effective_to` で有効期間を管理する。
部署廃止・統合が発生しても、過去の文書番号と所属履歴をそのまま追跡できる。
`merged_into_id` で統合先へのトレーサビリティを保持する。

### 社員所属の履歴テーブル

`employees.department_id` を直接持たず、`employee_departments` で期間付き履歴を管理する。
AD 同期時に異動履歴を保持でき、過去時点の所属状況を再現できる。

兼務は同一有効期間内に複数レコードを持つことで表現する。
`is_primary` で主務を区別し、同一有効期間内で `is_primary = true` のレコードは 1 件のみ（アプリ層で制約）。

文書登録時は主務所属部署をデフォルトとして表示し、兼務部署も含めた選択が可能。
選択した部署コードが `frozen_dept_code` に凍結される。

### プロジェクトと部署の間接紐づけ

プロジェクトは `department_id` を持たず、`discipline_id` 経由で部署と紐づく。
1 プロジェクトは 1 業務種別に属し、業務種別が担当部署を持つことで暗黙的に部署が決まる。

### 文書番号の凍結

`documents.frozen_dept_code` に登録時点の部署コードをコピーして凍結する。
部署再編後も文書番号を変更する必要がなく、過去の文書番号をそのまま参照できる。
