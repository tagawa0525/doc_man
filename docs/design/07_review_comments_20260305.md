## 総評

主要な構造は整理されているが、レビュー前は「採番・承認で必須の情報が `documents` に存在しない」ため、設計通りに実装しても破綻する状態だった。  
今回の修正で **ER / テーブル定義 / API / 採番仕様** の整合を取り、実装可能な最低条件を満たした。

---

## 指摘一覧（重要度順）

### 1) `documents` に文書種別 FK がなく採番根拠が欠落（Critical）

- **問題**: 採番は文書種別コードに依存するのに、`documents` が文書種別を保持していなかった。
- **影響**: 監査・検索・再採番判定を `doc_number` 文字列解析に依存し、整合性が壊れる。
- **対応**: `documents.doc_kind_id` を追加し、ER・テーブル・API・採番仕様を統一。

### 2) 文書ステータスの保存先が未定義（Critical）

- **問題**: 承認/回覧フローで `draft` などを定義しているが、DBカラムがなかった。
- **影響**: 状態遷移の永続化ができず、API仕様が実装不能。
- **対応**: `documents.status` を追加し、許容値チェックを定義。`05_approval_flow.md` に保存先を明記。

### 3) 採番 API 入力が曖昧（High）

- **問題**: `doc_type_code` という未定義パラメータ名を使用していた。
- **影響**: API・DB・採番処理のインターフェース不一致。
- **対応**: `doc_kind_id` に統一。

### 4) 同時採番時の衝突戦略が弱い（High）

- **問題**: トランザクション方針のみで、競合時の最終挙動が未定義。
- **影響**: 高負荷時に `409` が多発し、登録失敗率が上がる。
- **対応**: `UNIQUE(doc_number)` を最終防衛線とし、競合時最大3回リトライを採番仕様に追記。

### 5) `employee_departments.is_primary` が ER とテーブル定義で不一致（High）

- **問題**: ERには存在するがテーブル定義に欠落。
- **影響**: 主務所属を一貫して扱えない。
- **対応**: `03_tables.md` に `is_primary` を追加し、取得用インデックスと制約注記を追記。

### 6) APIに文書種別・文書台帳管理エンドポイントが欠落（Medium）

- **問題**: マスタは設計済みなのに運用APIが不足。
- **影響**: 初期データ投入・保守を手作業に依存。
- **対応**: `GET/POST/PUT /document-kinds` と `GET/POST/PUT /document-registers` を追加。

### 7) 文書番号サンプルが採番仕様と不一致（Medium）

- **問題**: API例で `CAD001-...` 形式が混在。
- **影響**: 実装時の誤解を誘発。
- **対応**: `内設計-2603001` 形式に統一。

### 8) 文書種別の連番桁数が空欄（Medium）

- **問題**: 仕様表に確定値がなく実装が止まる。
- **影響**: マイグレーション初期値を決められない。
- **対応**: 仕様表へ桁数を明記（`議=2`, その他=3）。

## 追加対応（GPT-5.3-codex レビューを受けての見直し）

### 9) 文書リビジョンが未定義（High）

- **問題**: `approval_steps.route_revision`（承認ルートの改版）は管理されていたが、文書内容自体の改版番号が存在しなかった。同一文書改版に対して承認ルートが複数回走るユースケース（承認者の差し替え等）も通常の運用で発生するため、2つの軸を独立して管理する必要がある。
- **影響**: 「文書改版 N に対して何回承認ルートが走ったか」が追跡できず、監査時に承認されたファイルの特定が困難。
- **対応**:
  - `documents.revision` を追加（登録時 `1`、文書内容修正時のみインクリメント）
  - `approval_steps.document_revision` を追加（ルート設定時点の `documents.revision` を凍結記録）
  - 05_approval_flow.md に「文書を修正した場合」「承認者だけ変えた場合」の再提出フローを整理

## 追加レビュー（Opus4.6 再編集後）

### 10) 承認APIの対象ステップ識別が曖昧（High）

- **問題**: `approve/reject` が `:step_order` 指定だと、`route_revision` 導入後に対象ステップを一意に決められない。
- **影響**: 旧ルートへの誤更新、承認競合、監査ログ不整合。
- **対応**: エンドポイントを `:approval_step_id` 指定へ変更し、最新 `route_revision` のアクティブステップ以外は `422` を返す仕様へ修正。

### 11) `doc_kind_id` 更新条件の判定が非実装的（Medium）

- **問題**: 「`approved/circulating` を経由していない」は履歴モデルなしでは厳密判定できない。
- **影響**: 実装依存の解釈差で API 挙動がぶれる。
- **対応**: 判定条件を「現在ステータスが `draft/rejected` かつ `approval_steps/circulations` レコード未存在」に変更。

### 12) `revision` をクライアント更新にすると競合しやすい（Medium）

- **問題**: クライアントが `revision + 1` を指定する方式は同時更新に弱い。
- **影響**: 競合時の失敗増加、UI再試行ロジックの複雑化。
- **対応**: `revision` は読み取り専用にし、文書内容変更時にサーバー側で自動インクリメントへ変更。

### 13) POST文書作成レスポンスの改版初期値が不明確（Low）

- **問題**: 201 応答に改版初期値が明示されていなかった。
- **影響**: クライアント実装で初期表示値の解釈差が出る。
- **対応**: `POST /documents` の 201 説明に `revision = 1` を追記。
