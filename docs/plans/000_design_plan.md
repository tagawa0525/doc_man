# 設計文書作成計画

## 概要

概念設計が固まったため、以下の設計文書を `docs/design/` 以下に作成する。
各文書の責務を明確に分離し、必要な文書だけを参照できるようにする。

---

## 作成する設計文書

### 1. `docs/design/01_overview.md` -システム概要・ドメイン構成

背景・目的・ドメイン一覧・スコープ外の整理。
概念設計の合意内容をそのまま文書化する。

### 2. `docs/design/02_er.md` -ER図

Mermaid によるエンティティ関連図。
テーブル名・主キー・外部キー・主要カラムを記載する。実装の直接の根拠とする。

### 3. `docs/design/03_tables.md` -テーブル定義

各テーブルのカラム・型・制約・インデックスを一覧する。
設計上の判断理由（なぜこの構造にしたか）もあわせて記載する。

### 4. `docs/design/04_document_numbering.md` -文書番号採番仕様

`{種別コード}{部署コード}-{YYMM}{連番}` のルール・採番手順・部署再編時の挙動を記述する。

### 5. `docs/design/05_approval_flow.md` -承認・回覧フロー仕様

段階承認のステータス遷移・承認ルートの定義方法・回覧の既読管理を記述する。

### 6. `docs/design/06_api.md` -API 設計

エンドポイント一覧・リクエスト/レスポンス例・エラー仕様を記載する。

---

## 合意済みの主要設計判断

実装時に参照するため、確定事項をここに記録する。

- **部署**: 階層ツリー。`effective_from/to` による有効期間モデル。`code` は採番用に不変
- **社員所属**: 期間付き履歴テーブルで管理。AD から現役データを同期。退職者はレコード保持
- **業務種別**: 技術領域マスタ（CAD・FEM・機械補修・電気工事など）。担当部署を持つ
- **プロジェクト↔部署**: 業務種別を経由して紐づく。プロジェクトは部署を直接持たない
- **文書番号の部署コード**: 文書に紐づく業務種別の担当部署コードを登録時に凍結
- **1文書 = 1業務種別**: 文書は必ず1つの業務種別に紐づく
- **段階承認**: 承認者を順序付きで登録し、順番に処理する
- **回覧**: 宛先ごとに確認日時を記録。未確認者を把握できる
- **機密グレード**: 公開 / 社内 / 部外秘 / 機密 の固定4段階
- **権限**: RBAC（admin / project_manager / general / viewer）

## 実施順序

1. 設計文書 01〜05 を作成・レビュー
2. 設計文書 06（API）を作成・レビュー
3. DB マイグレーションファイル作成（03 に基づく）
4. 実装（組織 → 業務種別 → プロジェクト → 文書 の依存順）
